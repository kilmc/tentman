# Tentman Compatibility And Preservation Philosophy

## Summary

Tentman should help developers build sites exactly how they like, then give non-technical editors a manageable, friendly way to update those sites.

That promise depends on trust. Tentman should not casually rewrite a developer's files, hide content in proprietary systems, or make old sites fragile. At the same time, Tentman is still pre-v1, and this is the moment to choose the best foundation rather than preserve draft decisions that only one current user would experience.

This document defines both sides of that philosophy:

- before the stable line, prioritize the best architecture
- after the stable line, treat compatibility and preservation as product values

## Product Belief

The web should have more great, durable, custom-built sites for artists, creators, small businesses, friends, family, and clients.

Developers should be able to build those sites with the tools and taste they prefer. Editors should be able to manage content without needing to learn Git, markdown, or deployment details.

Tentman exists between those two needs. It should respect both.

## Pre-V1 Foundation Policy

Tentman is not yet at the point where every current config shape must be preserved forever.

Right now, the project should optimize for the best long-term architecture:

- choose clearer data models over compatibility with early drafts
- remove awkward migration residue when it blocks a better design
- prefer a clean manifest shape over one contorted around temporary usage
- fix identity semantics before more features depend on them
- move shared behavior into core before web and CLI drift further apart

This is not a rejection of backwards compatibility. It is a recognition that the stable contract has not been declared yet.

Current Theresa/test-app usage is valuable as real product feedback, but it should not force a less elegant foundation just because one local draft config already exists. If a better shape requires updating those configs, update them.

At some future point, Tentman should intentionally declare a stable line. From that point forward, the preservation and compatibility commitments below become much stricter.

## Stable-Line Compatibility Policy

Once Tentman declares a stable config/content contract, existing sites should keep working for a long time.

Target principle:

> A site built with Tentman years earlier should still load, be understandable, and be manageable.

That does not mean every old site automatically receives every new feature. It does mean Tentman should avoid breaking existing supported features without a clear migration path.

Stable-line commitments:

- old supported configs continue to parse
- old supported manifests continue to load
- old supported content remains editable
- new features are additive where practical
- breaking changes require explicit migrations
- unsupported future schemas are not written by older runtimes
- risky writes are blocked or warned before they happen

## Preservation Policy

Tentman should be a polite guest in a developer's repo.

Normal editing should preserve:

- file locations
- content structure
- developer-owned fields
- existing formatting where practical
- project-local conventions
- site runtime assumptions

Tentman-owned metadata can be added when needed, but it should be clearly identifiable and stable.

Examples of Tentman-owned metadata:

- `_tentmanId`
- generated navigation state
- draft asset references
- future generated indexes or manifests

Developer-owned values should remain separate:

- slugs
- titles
- labels
- filenames
- routes
- custom frontmatter fields
- site-specific content fields

## Formatting Philosophy

Tentman should not constantly reformat files just because a developer opted into using it.

Preferred direction:

- normal saves should preserve surrounding formatting where possible
- narrow edits should produce narrow diffs
- full normalization should be explicit
- formatting drift should be inspectable before editors encounter it
- developer formatting preferences should be respected where possible

CLI commands should support this:

```sh
tentman format --check
tentman format --write
```

`format --check` tells a developer what Tentman would currently rewrite.

`format --write` lets a developer intentionally normalize files in a separate commit.

Long term, Tentman should rely less on whole-document serialization for ordinary edits, especially for markdown frontmatter. Where safe, it should patch the specific changed field and preserve the rest of the document.

## Stable Identity Philosophy

Stable ids are system metadata, not content.

Rules:

- stable ids are generated by Tentman
- stable ids are not derived from slugs, filenames, labels, or titles
- stable ids are consistently shaped
- stable ids survive slug/title/filename changes
- stable ids are not exposed as ordinary author-facing fields

Preferred provisional prefix:

```txt
tent_
```

Example:

```txt
tent_01HX7Q9E4K8F2G6M9P3A0B1C2D
```

The prefix is intentionally visible so developers can tell that the value is Tentman-owned. `tmid_` remains a possible alternate if `tent_` feels too cute or unclear in practice.

## Navigation Manifest Philosophy

The navigation manifest should be useful to both Tentman and the site.

It should not require each consumer site to rebuild navigation through fragile joins against Tentman internals. If a site wants to render navigation from the manifest, the manifest should carry enough materialized information to make that reasonable.

Pre-v1, prefer a clean manifest foundation over preserving early `items: string[]` shapes.

Desired qualities:

- stable ids for identity
- labels for display
- slugs for route semantics where useful
- hrefs when Tentman can derive them safely
- groups that are directly renderable
- refresh/rebuild commands that keep materialized data current

The manifest can be generated from config and content, but once generated it should be legible and practical for a developer to use.

## Core Ownership Philosophy

Behavior that defines "what Tentman would do" should live once.

Core should own:

- config discovery
- path resolution
- content parsing
- content writing
- stable ids
- navigation refresh/rebuild
- formatting checks
- schema compatibility
- asset validation

The web app, CLI, tests, and agents should call core. They should not each implement their own interpretation of Tentman rules.

This protects developers from inconsistent behavior and protects the product from subtle drift.

## Free Core Value And Paid Product Value

Tentman's basic usefulness should be broadly available.

The core idea is not to trap content behind a paid product. The value should come from making dev-built sites manageable without forcing people into Wix, Squarespace, or a heavy headless CMS.

Paid value can come later from:

- hosted convenience
- collaboration
- preview/publishing workflows
- Git provider integrations
- deployment integrations
- team permissions
- media handling
- richer extension infrastructure
- support and confidence

The internal core package can remain private for now as a distribution choice, but the product should still be designed around portable content and understandable config.

## Future Stable Contract

When Tentman is ready to declare a stable line, it should include:

- an explicit schema/version policy
- supported compatibility range
- migration commands
- write-safety rules
- documented deprecation process
- tests for old fixture sites

Until then, the project should move decisively toward the best version of itself.
